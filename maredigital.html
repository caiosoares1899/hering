Visão geral

O que é o Maré Digital

Kanban ágil single-file que roda no browser. Deploy = git push no GitHub Pages.

📄
Single-file app
HTML, CSS e JS num único arquivo por tela. Sem build, sem dependências locais, sem servidor próprio.
🔄
Tempo real
Firebase Realtime DB via WebSocket — qualquer mudança aparece no board de todos em menos de 1s.
🤖
Agente de IA
Lê o estado atual e age sobre ele: move cards, cria tarefas, atribui donos, analisa métricas.
🏢
Multi-squad
Um deployment serve N squads via ?squad=id. Dados isolados por squad no Firebase.
📱
PWA + Mobile
Instalável no Android/iOS. Bottom nav, touch drag-and-drop, safe-area. Service Worker para cache offline.
🔒
Acesso restrito
Login exclusivo Google @ciahering.com.br. Domínio validado no servidor via Firebase Rules.

Infraestrutura

Arquitetura

Quatro peças externas, nenhum servidor próprio.

📦
O que fica no Firebase
Cards, tags, colunas, sprint cfg, automações, backups, comentários, logs de erro, presença. Segregado por squad.
🔑
O que fica no Worker
A chave da Anthropic como secret Cloudflare. Browser nunca a vê. Worker pina model e limita tokens.

kanban.html

Board & Visualizações

Três modos de visualização, filtros multicritério, drag-and-drop mouse e toque.

FunçãoO que faz
renderNormal()Grid de colunas. Cards ordenados por prioridade (critical→high→medium→low). Colunas minimizáveis.
renderRaiaOwner()"Raia · pessoa" — faixa horizontal por responsável. Sem dono = faixa de aviso pulsante no fim.
renderRaiaTag()"Raia · tipo" — faixa por tag. Cor da faixa = cor da tag. Sem tag = "Sem tipo".
toggleRaia()Cicla: Off → Raia pessoa → Raia tipo → Off. Rótulo do botão muda para indicar o modo.
passesFilter()Dono, tag, prazo (atrasado/hoje/7 dias), prioridade, impedido, com riscos — AND entre filtros.
handleDrag*()Drag-and-drop desktop com ghost-line indicando posição. Drop aciona Firebase + automações.
addTouchDnD()Touch DnD mobile via touchstart/move/end. Clone visual segue o dedo; detecta coluna via elementFromPoint().
openSearch()Busca por texto em título, descrição e PO. Navegação por teclado (↑↓ Enter). Abre card diretamente.
💤
Card aging
7 dias sem edição: desbotado. 14 dias: emoji 💤 + mais transparente. Sinal visual de estagnação.
🔗
Dependências
Cards vinculados via "depende de". Mini-mapa de dependências no card. Modal com árvore completa.
💬
Comentários
Thread por card com @menção a membros e @card. Persiste em squads/{id}/cards/{cid}/comments.
📊
Métricas de WIP
WIP, throughput, bloqueios e capacidade exibidos em tempo real. O Agente Ágil os lê e interpreta.

kanban.html

Sistema de Cards

Objeto rico com campos ágeis completos. Auto-save, OKR flag, anexos e dependências.

estrutura de um card
{ id:"c1749123", col:"progress", tag:"data", owner:"CS", participants:["CS","RP"], due:"2025-06-20", priority:"high", sp:"5", desc:"...", po:"Objetivo SMART...", // campo exclusivo do PO checklist:[{t:"item",done:false}], riscos:["Dep ERP"], okr:false, dependsOn:"c1749000001", // card pai blocker:false, archived:false, created:"Caio", edited:"Caio", createdAt:"2025-06-01", editedAt:"2025-06-08" }
🎯
Insights do PO
Campo exclusivo para objetivo, contexto e critério de aceite SMART. Isolado da descrição técnica.
🏆
Flag de OKR
Card marcado como OKR da sprint. Painel agrega OKRs de todos os squads em uma visão consolidada.
📎
Anexos de links
Links dentro do card com detecção automática do serviço (Jira, Figma, GitHub, Drive etc.) e favicon.
💾
Auto-save
Debounce 800ms: sem nova edição no modal, salva automaticamente no Firebase.
🚥
Semáforo de prazo
Verde/amarelo/vermelho. Cards em "Concluído" congelam o badge com ✓ — nunca ficam vermelhos.
📋
Modelos reutilizáveis
Salva tag, PO, checklist zerado e riscos. Disponível em Toolbar → Modelos para criar em qualquer sprint.

IA · kanban.html

Agente Ágil

IA que lê o board e age sobre ele. Disponível para PO e Organizadores.

🧠
Contexto dinâmico — agCtx()
A cada mensagem monta snapshot completo: todos os cards com ID exato, coluna, dono, prazo, checklist, OKR, riscos, PO — mais WIP, capacidade, sprint e membros. Enviado como system prompt. Agente sempre vê o estado mais recente.
Ferramenta (tool_use)O que faz
mover_cardMove para qualquer coluna. Aceita ID exato ou título parcial.
atualizar_prazoRedefine data de entrega (YYYY-MM-DD).
marcar_checklistMarca/desmarca item pelo índice.
criar_cardCria card completo (prioridade, PO, dono, prazo, tag).
atualizar_ownerAtribui responsável. Resolve "eu", "me" e nomes para iniciais.
definir_prioridadeDefine ou remove prioridade (critical/high/medium/low).
salvar_modeloSalva como modelo reutilizável na biblioteca.
salvar_recorrenteAdiciona à lista de recorrentes.
excluir_cardExclui permanentemente. Requer confirmado: true.
atualizar_descricaoEdita título, descrição ou PO de qualquer card.
📊
Log de uso
Cada chamada registra timestamp, usuário, aba e primeiros 120 chars da pergunta em squads/{id}/agent_usage. O Painel exibe frequência de uso por squad.

kanban-dev.html

Sandbox & Dev Seed

Ambiente isolado para testar features novas antes de ir para produção.

🧪
O que é o kanban-dev
Cópia funcional do kanban.html com squad padrão em dev. Dados isolados em kanban/squads/dev/ — não interferem em produção. Mesmo Firebase, mesmo Worker, mesma lógica.
🌱
Dev Seed — runDevSeed()
Se o board dev está vazio ao abrir, um overlay de seed lista os cards que serão criados. Ao confirmar, popula o board com dados fictícios realistas — cards com tags, donos, prazos, checklists e riscos. Log visual em tempo real mostra cada card criado. Ideal para demos e testes de UI.
🔁
Ciclo de desenvolvimento
Feature → testada no dev com seed → validada → portada para kanban.html cirurgicamente (sem sobrescrever o arquivo).
📋
Squad dev no Painel
Aparece no seletor do Painel identificado como sandbox. Backups do dev são separados dos de produção.

Painel — Visão Geral

Dashboard de gestão para diretoria. Lê dados de todos os squads simultaneamente.

🔭
O que o Painel faz que o Kanban não faz
O kanban foca em um squad por vez. O painel lê todos os squads simultaneamente e entrega: visão consolidada de OKRs, blockers e riscos; calendário unificado; log de erros de todos os boards; gestão global de usuários; Board Setup para criar novos squads; backup global em um clique.

Cria novos squads dinâmicos sem editar código.

🏗
Como funciona
ADM preenche nome, emoji, cor e ID. Painel salva em kanban/squads_meta/{id}. Na próxima abertura do kanban.html, loadSquadsFromFirebase() carrega o squad e o adiciona ao SQUAD_META_LIVE — disponível via ?squad=id. Preview em tempo real durante a criação.
🌍
Backup global
Um clique exporta o JSON de todos os squads ou de cada um individualmente. Configuração de e-mail automático separada do backup por squad.
🔧
Config de squad no painel
Sprint, WIP, objetivo e colunas visíveis de cada squad configuráveis diretamente do painel, sem precisar abrir o kanban.

Segurança

Camadas de segurança — v8.3

Seis camadas independentes. Se uma falhar, as outras continuam de pé.

✓ ativo
Escape de HTML (XSS)
esc() e safeUrl() aplicados em todos os 35 pontos de renderização. Dados do Firebase nunca entram crus no innerHTML. Presente em kanban, kanban-dev e painel.
✓ ativo
Firebase Database Rules
Verifica auth.token.email no servidor. Campo role só alterável por PO. Presença: cada user escreve só o próprio nó.
✓ ativo
Worker + Origin Validation
Worker v3.0 valida header Origin contra ALLOWED_ORIGINS. Sem variável → retorna 500 (fail-secure). Bloqueia uso indevido da URL.
✓ ativo
Chave API protegida
Chave da Anthropic como secret Cloudflare. Nunca no HTML, nunca em localStorage, nunca em trânsito para o browser.
✓ ativo
RBAC + ADMs hardcoded
PO/Org/Membro com permissões distintas. ADMs via ADM_EMAILS têm papel mínimo PO — não rebaixável editando o banco.
✓ ativo
Model fixo + max_tokens
Worker pina claude-sonnet-4-20250514. Teto de 2000 tokens por requisição. Cliente não controla nem o modelo nem os tokens.

Firebase Database Rules

Aplicadas no servidor — não há como contornar pelo browser.

firebase-rules.json
// squads/{id}/dados — qualquer @ciahering autenticado lê e escreve "squads": { "$squadId": { "dados": { ".read": "auth != null && auth.token.email.endsWith('@ciahering.com.br')", ".write": "auth != null && auth.token.email.endsWith('@ciahering.com.br')" }}}, // usuarios/$uid — usuário escreve o próprio; PO escreve qualquer "usuarios": { "$uid": { ".write": "auth.uid === $uid || root.child('kanban/usuarios/'+auth.uid+'/role').val()==='po'", "role": { // só PO muda; 1ª gravação como 'membro' sempre permitida ".validate": "root.child('kanban/usuarios/'+auth.uid+'/role').val()==='po' || newData.val()===data.val() || (!data.exists() && newData.val()==='membro')" } }}, // presence/$uid — cada user só escreve o próprio heartbeat "presence": { "$uid": { ".write": "auth.uid === $uid && auth.token.email.endsWith('@ciahering.com.br')" }}
